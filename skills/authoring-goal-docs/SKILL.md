# Authoring Goal Docs

Create a durable statement of product intent, not an implementation plan.

## Workflow

1. Read the repository's contribution guidance and existing goal-document conventions.
2. Research the relevant code, docs, task history, and version history. Delegate independent scans when the surface is large.
3. Treat every existing claim as evidence to confirm, not as truth to preserve.
4. Synthesize the durable goal, scope, non-goals, constraints, risks, and definition of done.
5. Before writing, present the reconstructed intent and ask only the questions that require product-owner judgment.
6. Write the goal document plus a separate tracker for open work when the project convention supports one.
7. Link the goal from the project's index and verify links and stated facts.

## Document Boundary

Keep durable intent separate from current execution state:

- Goal: what success means and why it matters.
- Context: decisions and rationale likely to prevent future re-litigation.
- Task tracker: open work and concise history.
- Status: a replaceable current snapshot.

Do not turn the goal into a detailed implementation plan. Hand implementation planning to `gap-analysis` after the goal is agreed.
